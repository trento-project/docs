= Trento Documentation
ifndef::site-gen-antora[:relfileprefix: /content/]

This repository is the central documentation source for the Trento project.

== Repository Overview

=== Repository Directory Layout

==== Documentation Content

* link:https://github.com/trento-project/docs/tree/main/trento[`trento`] - user-facing Trento documentation used for both the docs hub and the official SUSE documentation
* link:https://github.com/trento-project/docs/tree/main/content[`content`] - project documentation such as architecture, ADRs, coding standards, RFCs, templates, and internal notes

==== Build, UI, and Publishing Tooling

* link:https://github.com/trento-project/docs/tree/main/trento-docs-site[`trento-docs-site`] - Antora playbook, extensions, and site build logic
* link:https://github.com/trento-project/docs/tree/main/trento-docs-site-ui[`trento-docs-site-ui`] - custom Antora UI bundle
* link:https://github.com/trento-project/docs/tree/main/.github/workflows[`.github/workflows`] - CI and publishing workflows for documentation validation and site builds

=== How This Repository Works

This repository combines multiple documentation sources into the final published outputs.

* `trento/` provides the user-facing Trento documentation
* `content/` provides the project documentation maintained in this repository
* upstream Trento component repositories provide additional component documentation for the docs hub

The Antora site is assembled from these sources with the playbook and extensions in `trento-docs-site/`.
The custom UI for the docs hub is maintained in `trento-docs-site-ui/`.
The CI for the user-facing documentation intentionally mirrors link:https://github.com/SUSE/doc-unversioned[SUSE/doc-unversioned] with small Trento-specific adjustments.
This keeps the `trento/` sources compatible with both the Trento docs hub build and the SUSE documentation toolchain.

== Documentation Sources

=== User-Facing Documentation

The link:https://github.com/trento-project/docs/tree/main/trento[`trento`] directory is the single source for Trento user-facing documentation.

=== Project Documentation

This repository contains project documentation maintained in this repository, including:

* xref:architecture/trento-architecture.adoc[Architecture]
* xref:architecture/adr/README.adoc[Architecture Decision Records (ADR)]
* xref:coding-standards/README.adoc[Coding standards]
* link:https://github.com/trento-project/docs/tree/main/content/rfc[Request for Comments (RFC)]
* link:https://github.com/trento-project/docs/tree/main/content/internal-notes[Internal Notes]
* link:https://github.com/trento-project/docs/tree/main/content/templates[Templates]

These sources are maintained directly in this repository.

=== Component Documentation from Upstream Repositories

During the Antora build, this repository fetches documentation directly from upstream Trento project repositories and assembles it into the docs hub under the `Components` section.

These sources usually live in the upstream repositories under `docs/` or `guides/`.
For example:

* link:https://github.com/trento-project/web/tree/main/guides[`trento-project/web/guides`]
* link:https://github.com/trento-project/agent/tree/main/docs[`trento-project/agent/docs`]

The build process adapts this upstream content to the Antora layout used by the docs hub.
For example, documentation maintained in an upstream repository such as link:https://github.com/trento-project/web[`trento-project/web`] is published in the docs hub as link:https://www.trento-project.io/docs/web/README.html[`/docs/web/README.html`].
The source of truth still remains the upstream repository.

[IMPORTANT]
====
All changes to component documentation must be made in the source repository.
They are picked up and included in the next Antora build of link:https://www.trento-project.io/docs/[trento-project.io/docs].
====

== Published Outputs

=== Docs hub

The Trento docs hub is published at link:https://www.trento-project.io/docs/[trento-project.io/docs].
It is built with https://antora.org/[Antora] from the user-facing documentation in `trento/`, the project documentation in `content/`, and the fetched component documentation from upstream repositories.

=== Official SUSE documentation

The official SUSE documentation is published at link:https://documentation.suse.com/sles-sap/trento/html/SLES-SAP-trento/index.html[documentation.suse.com].
It is generated from the same user-facing documentation sources in link:https://github.com/trento-project/docs/tree/main/trento[`trento/`].

=== Published Versions

The published documentation currently exposes two versions:

* `main` - the pre-release version, reflecting the latest state of the project, published at link:https://www.trento-project.io/docs/index.html[`/docs/index.html`]
* `latest` - the current release version, published at link:https://www.trento-project.io/docs/latest/index.html[`/docs/latest/index.html`]

No older published versions are currently maintained in this repository.

== Local Development

=== Where To Edit

Use this quick guide to decide where documentation changes belong:

* Edit link:https://github.com/trento-project/docs/tree/main/trento[`trento/`] for Trento user-facing product documentation.
* Edit link:https://github.com/trento-project/docs/tree/main/content[`content/`] for project documentation such as architecture, ADRs, coding standards, RFCs, templates, and internal notes.
* Edit the documentation sources in the upstream component repository for component documentation published under `Components` in the docs hub.
+
Examples:
+
** link:https://github.com/trento-project/web/tree/main/guides[`trento-project/web/guides`]
** link:https://github.com/trento-project/agent/tree/main/docs[`trento-project/agent/docs`]

=== Preview the Trento docs hub

From the repository root, build the docs site and serve it on port 3000:

[source,bash]
----
docker compose up trento-docs
----

Open http://localhost:3000 while the command is running.

The generated HTML is in `trento-docs-site/build/trento-docs-site-public`.

=== Preview SUSE documentation

This preview uses a DAPS container maintained by the documentation team.
It contains the official build toolchain used for the SUSE documentation output, so you can locally preview the same Trento user-facing sources both for the docs hub and for the official documentation published on the SUSE documentation site.

From the repository root, you can preview either the HTML output or the PDF output.

==== Preview SUSE documentation HTML

To build the DocBook HTML in a container and serve it on port 3001:

[source,bash]
----
docker compose up daps-html-serve
----

Open http://localhost:3001 while the command is running.

The generated HTML is in `trento/build/SLES-SAP-trento/html/SLES-SAP-trento`.
Each run clears `trento/build/SLES-SAP-trento` before rebuilding to avoid stale files.

==== Preview SUSE documentation PDF

To build the PDF output in a container and serve it on port 3002:

[source,bash]
----
docker compose up daps-pdf-serve
----

Open http://localhost:3002/SLES-SAP-trento_en.pdf while the command is running.

The generated PDF is in `trento/build/SLES-SAP-trento/SLES-SAP-trento_en.pdf`.
Each run clears `trento/build/SLES-SAP-trento` before rebuilding to avoid stale files.

=== Preview everything

Start both preview servers with one command:

[source,bash]
----
docker compose up
----

To start all services in the background and skip compose logs:

[source,bash]
----
bash dev-docs-env.sh
----

If your local Docker setup requires elevated permissions, run it with `sudo`.

Once it finishes, open:

* http://localhost:3000 (Trento docs hub)
* http://localhost:3001 (SUSE HTML build)
* http://localhost:3002/SLES-SAP-trento_en.pdf (SUSE PDF build)

== Contribution Notes

For docs under `content/`, keep image files in `content/images/`.

This setup renders images correctly in both GitHub preview and Antora.

In pages that include images, add this fallback directive:

[source,asciidoc]
----
\ifndef::imagesdir[:imagesdir: ../images]
----

Then reference images relative to `imagesdir`, for example:

[source,asciidoc]
----
image::release-branching.png[]
image::analytics/create-container.png[]
----

See a real example in
link:https://github.com/trento-project/docs/blob/main/content/internal-notes/analytics-setup.adoc[`content/internal-notes/analytics-setup.adoc`].

The collector scan preserves subdirectories from `content/images/`; it does not flatten files.
